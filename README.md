## 🧠 The Problem & Solution

**The Problem:** Structuring a scalable React application is tedious. Developers often waste hours mapping out component hierarchies, prop-drilling paths, and file structures before writing a single line of code.

**The Solution:** This application acts as a Senior Architecture pair-programmer. You describe the feature (e.g., *"A micro-frontend e-commerce cart sidebar"*), and the engine generates a strictly typed, recursively nested JSON tree. It enforces Atomic Design principles, stores the architecture in a PostgreSQL database, and renders it in an interactive, mutable UI.

---

## ✨ Core Features

* **LLM-Driven Structural Generation:** Prompts are processed using Vercel's AI SDK and Groq's high-speed inference to guarantee structured JSON output.
* **Interactive Recursive UI:** Custom React components that dynamically parse and render deeply nested self-referencing data structures.
* **Optimistic UI Updates (CRUD):** Users can edit node names, update descriptions, or prune entire branches. UI updates happen instantly, with background synchronization to the database.
* **Session & State Management:** Secure, middleware-protected routing using Supabase Auth (Google OAuth) with Server-Side Rendering (SSR) cookie validation.
* **Edge-to-Edge SaaS Layout:** A modern, flexbox-driven workspace featuring a history sidebar and a floating command palette.

## 🏗️ System Architecture

The application follows a modern Next.js 15 App Router paradigm, heavily utilizing Server Components for data fetching and Server Actions for mutations.

### Tech Stack
* **Framework:** Next.js 15 (React 19)
* **Styling:** Tailwind CSS + Lucide Icons
* **Database:** Supabase (PostgreSQL)
* **Authentication:** Supabase Auth (Google SSO + SSR Middleware)
* **AI Provider:** Groq (`llama-3.3-70b-versatile`) via Vercel AI SDK

### Database Schema (Relational Integrity)
The database relies on two primary tables with strict relational rules to ensure data integrity without application-level overhead.

1. `projects`: Tracks the overarching prompt and user session.
2. `nodes`: Stores the individual atomic components. 

**Key Design Choice: Self-Referencing Foreign Keys & Cascade Deletes**
The `nodes` table utilizes a `parent_id` column that references its own table's `id` column. This allows for infinite recursion (e.g., Atoms inside Molecules inside Organisms). 
Both `project_id` and `parent_id` utilize SQL `ON DELETE CASCADE`. If a user deletes a root "Page" node via the UI, PostgreSQL automatically prunes the entire multi-level branch of children instantly, requiring zero recursive API calls.

## 🛠️ Key Technical Decisions & Reasoning

#### 1. Why Groq and Llama 3?
Generating massive nested JSON objects requires immense token output speed. Traditional LLM APIs resulted in high latency, causing poor UX. Groq's LPU architecture combined with Llama 3.3 70B guarantees near-instant structured JSON generation, keeping the user in a state of flow.

#### 2. The UUID Mapper (Data Persistence)
LLMs cannot generate valid PostgreSQL UUIDs reliably. To maintain the parent-child relationships generated by the AI, the `/api/generate` route implements an **ID Dictionary Mapper**. It intercepts the LLM's mock IDs (e.g., `org-1`, `atom-3`), generates real cryptographic UUIDs on the server, maps the connections safely, and performs a single batch insert to Supabase.

#### 3. Optimistic Rendering vs. Pessimistic Writes
To make the application feel like a native desktop tool, the Tree UI employs Optimistic Rendering. When a user edits or deletes a node, local React state is mutated instantly. The actual `PATCH` or `DELETE` request is fired asynchronously in the background. 

#### 4. Server Actions vs API Routes
Authentication flows and database deletions were migrated to Next.js Server Actions (`'use server'`). This drastically reduced client-side bundle size, eliminated the need for complex `fetch` wrappers, and ensured secrets (like Supabase Service keys) never leaked to the browser.

## 💻 Local Setup & Development

To run this project locally, you will need Node.js, a Supabase project, and a Groq API key.

**1. Clone the repository:**
```bash
git clone [https://github.com/YOUR_USERNAME/frontend-planner.git](https://github.com/YOUR_USERNAME/frontend-planner.git)
cd frontend-planner


npm install
```
## Set up Environment Variables:
Create a .env.local file in the root directory and add the following:

``` bash 
NEXT_PUBLIC_SUPABASE_URL=your_supabase_project_url
NEXT_PUBLIC_SUPABASE_ANON_KEY=your_supabase_anon_key
GROQ_API_KEY=your_groq_api_key

```
## Run the development server:
``` bash 
npm run dev
```

Visit http://localhost:3000 to view the application.
